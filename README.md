# Mao Thought Skill

这个仓库现在收敛为一个核心目标：把《毛泽东选集》蒸馏成一个可迁移的方法型 skill。

重点不是模仿旧文风，也不是摘录语录，而是保留毛选最有辨识度的方法，并做成“主文件承载核心思想 + 三个场景补充”的结构：

- 矛盾分析法
- 调查研究
- 群众路线
- 战略战术思维
- 区分不同性质矛盾
- 从判断走到办法和组织动作

## 使用方式

默认必须先读取：

- [SKILL.md](SKILL.md)
- [references/source-map.md](references/source-map.md)

这两个文件共同构成这个 skill 的基础加载单元。少了 `source-map.md`，就不算完整加载。

在此基础上，再按任务补充一个场景文件：

- 大会讲话：`references/scene-conference-speech.md`
- 向上汇报：`references/scene-report-upward.md`
- 问题分析探讨：`references/scene-issue-analysis.md`

也就是说，`source-map.md` 不再是“需要时再读”的文件，而是每次启动都要默认生效的核心文件。

## 推荐调用

- “用 `$mao-thought-conference-speech` 写一篇年度大会讲话，重点讲当前关口和下一步抓手。”
- “用 `$mao-thought-report-upward` 写一份向上级的汇报，先讲态势，再讲问题和思路。”
- “用 `$mao-thought-issue-analysis` 分析一下这个平时争论的问题，抓主要矛盾和几种思路。”

# 一期验收冲刺会议主题稿

各位同事：

今天这个会，要先把局面讲清楚。

我们的项目已经进入一期验收前的最后阶段。大盘基本定了，主体功能已经搭起来了，前一段时间该攻的难点也已经攻下了一批。应该说，路子没有走偏，基础不是没有，队伍也不是不能打。问题不在于项目是不是做不成，而在于到了最后关口，剩下这 10% 的工作，恰恰是最不能含糊、最不能拖、最容易影响验收结论的 10%。

现在摆在面前的是一个很具体的情况：原本预计还需要一个月完成的工作量，现在必须压缩到 15 天以内完成。这不是一般意义上的赶进度，这是阶段转换了。项目从前期建设阶段，转入了集中收口、集中联调、集中验收准备的冲刺阶段。这个阶段如果还按平时的节奏干，还按平均用力的办法推，还按“差不多”“先往前做着看”的习惯来处理，最后就一定会在验收前暴露出一串问题。

所以，今天这个会的主题只有一句话：统一认识，集中兵力，拿下项目一期验收前最后 15 天的冲刺任务。

先讲一点成绩。这个时候讲成绩，不是为了给自己宽心，而是为了把底气讲清楚。我们已经具备冲刺的条件。

一是主线是清楚的。核心目标、主要场景、总体方向没有跑偏，这说明前一阶段不是白干。

二是基础是存在的。系统框架、主体模块、主要流程都已经在了，现在不是从头开工，而是做最后收口。

三是队伍是经过考验的。前面硬仗打过来了，说明这支队伍有执行力，也有攻坚能力。

但越是这个时候，越要把问题点透。当前最带全局的问题，不是“还剩 10% 没做完”这个表面现象，而是剩余任务和剩余时间已经不对称了。说得再直一点，不是任务单纯重了，而是原来按月推进的工作，现在必须按天组织；原来可以往后顺延的事情，现在必须在验收节点前闭环；原来局部消化的问题，现在会直接变成全局风险。

这就是当前的主要矛盾：验收节点已经压到眼前，而我们的剩余任务组织方式、推进节奏和协同办法，还没有完全切换到冲刺状态。

这个矛盾的性质，要看准。它不是哪个人偷懒的问题，不是哪一个模块单独掉队的问题，也不是前期工作整体失败的问题。它本质上是一个冲刺阶段的组织问题，是一个节奏重排问题，是一个兵力集中问题。性质看不准，办法就会跑偏。要么变成泛泛加压，人人喊忙；要么变成互相埋怨，个个自保；要么表面热闹，关键路径却没有真正压住。

看清这一点，下面的工作就好定了。总的要求是：一切服从一期验收，一切围绕最后 10% 的关键功能闭环，一切围绕 15 天内形成可验收状态来组织。

围绕这个总要求，我讲四点。

第一，任务要重排，兵力要集中。到了今天，不能再平均用力，不能再眉毛胡子一把抓。凡是直接影响一期验收结论的功能、联调、稳定性、数据、材料，全部提到最前面来。凡是不影响本次验收通过、可以顺延到后续优化的内容，坚决往后放。现在不是比谁做得多，而是比谁先把最要命的地方打下来。

第二，关键路径要拉直，卡点要全部翻出来。每个负责人都要把自己手里的剩余项讲清楚，讲清楚还差什么，卡在什么地方，依赖谁，什么时候必须交，什么状态才算完成。现在最怕的不是有问题，最怕的是问题藏在水下，大家各自认为“应该差不多”，最后临门一脚一起炸。今天会后，剩余任务必须形成清单，关键依赖必须形成表，风险项必须逐条挂账。

第三，推进节奏要改成按天作战。项目到了这个阶段，不能再按周看，必须按天推进、按天检查、按天销账。今天定的事，明天就要看到结果；明天解决不了的，后天必须有升级处理。项目负责人要盯住总盘子，模块负责人要盯住自己的闭环，产品、开发、测试、实施要在一张作战图上协同，而不能各干各的。冲刺阶段的快，不是谁冲得快，而是整条链路都不卡。

第四，开发收口、联调验证、验收准备必须并行推进。不能再抱着一种想法：等代码都写完了再测试，等功能都补齐了再准备验收材料。到了最后关口，这种打法一定来不及。开发往前赶，测试要同步介入；功能在闭环，材料也要同步整理；问题一暴露，马上拉人处理，不能往后堆。我们不是等所有事情“自然完成”，而是要有组织地把它们压到同一个节点上。

这里还要补一句。现在这个时候，既不能盲目乐观，也不能自己吓自己。盲目乐观，会把风险看轻；自己吓自己，会把队伍打散。正确的态度就是两条：一条是清醒，知道时间紧、任务硬、容错空间很小；一条是坚定，知道我们不是没基础，不是没办法，也不是拿不下来。

接下来的 15 天，真正考验我们的，不只是编码能力，而是判断能力、组织能力、协同能力。谁能把主要矛盾抓住，谁能把兵力摆到关键处，谁能把每天都变成有效推进，谁就能把这场仗打下来。

最后，我提三点要求。

第一，今天会后，各负责人立即重新梳理本模块任务，不允许再沿用原来的松散推进表。

第二，今晚之前形成剩余任务清单、关键路径清单、风险清单，做到任务到人、节点到天、标准到项。

第三，从明天开始，进入冲刺机制，按天通报、按天协调、按天销账，直到一期验收前最后一个节点。

时间已经不宽裕了，话也不必说得太满。我们现在要做的，不是再讨论难不难，而是把最后这 15 天真正组织起来，把原来一个月的工作，打成一场重点突出、节奏明确、责任压实的集中攻坚。

这最后一段，必须拿下。
